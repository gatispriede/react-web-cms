Folder reorg — four architecture options

Lightest → heaviest reorganisation. Each keeps the build green; they differ in how much config you're willing to touch.

directory new/renamed unchanged convention-bound path comment

Option A — Minimal No config changes

Reorganise only src/frontend/components/. Server, Interfaces, enums, Api at root are untouched.
src/
├── Api/                     # untouched
├── Interfaces/              # untouched
├── Server/                  # untouched
├── enums/ constants/ …      # untouched
└── frontend/
    ├── pages/ public/ gqty/   # Next convention — untouched
    └── components/
        ├── admin/               # AdminApp, settings, config inputs
        ├── public/              # public-site only (Logo, top-bar…)
        ├── sections/            # per-module folders
        │   └── hero/
        │       ├── Hero.tsx
        │       ├── Hero.editor.tsx
        │       ├── Hero.scss
        │       └── index.ts
        ├── shared/              # ContentManager, SectionContent, wrappers
        └── ui/                  # dialogs, Logo, RichTextEditor
Cost~1 focused day. One subtree moves, no config touched.
Trade-offRoot-level sprawl (Server, Api, Interfaces, enums) remains.
Good forFastest legibility win; stepping-stone toward Option C later.

Option B — features / ui / lib split Frontend-only rewire

Standard "features / ui / lib" layout inside src/frontend/. Next-convention dirs stay put.
src/frontend/
├── pages/ public/ next.config.js  # Next convention — untouched
├── app/             # AppShell, providers
├── features/
│   ├── admin/         # AdminApp + settings tabs
│   ├── auth/         
│   ├── sections/      # per-module folders
│   └── site/          # public shell, top-bar, mobile nav
├── ui/              # shared AntD compositions, Logo, dialogs
├── lib/             # i18n, theme engine, gqty glue
├── api/             # domain gateways (Mongo/User/Asset…)
└── styles/          # scss tokens + globals
Cost~2 days: bulk git mv + import-path rewrite across frontend.
Trade-offServer + shared types at root still untouched (handle separately).
Good forReaders who recognise "features/ui/lib" instantly. No Next config risk.

Option C — Single /app tree Whole-repo reshape

Everything the running product needs under app/; everything that ships it under infra/; writing under docs/.
app/                        # the running product
├── web/                      # Next.js project (was src/frontend/)
│   ├── pages/ public/ next.config.js
│   ├── features/ ui/ lib/ api/ # Option B's internals
│   └── styles/
├── server/                   # standalone GraphQL (was src/Server/)
│   ├── routes/
│   ├── services/           # UserService, NavigationService…
│   ├── infra/              # mongo/redis connection, fileManager
│   └── schema.graphql
└── shared/                   # types + enums used by both
    ├── interfaces/
    ├── enums/ constants/
    └── api-dto/            # was src/Api/

infra/                      # deployment
├── docker/                 # Dockerfiles, compose.yaml
├── scripts/                # was Scripts/
└── certificates/

docs/                       # ROADMAP, THEMING, roadmap/
package.json next-env.d.ts  # forced at root
Cost~half a day of config + ~1 day rewire: Dockerfiles, compose.yaml, gqty.config.ts, i18next config, server entrypoint paths.
Trade-offBiggest diff. Every import path inside the repo shifts.
Good forSelf-explanatory top level: code / deployment / writing. Newcomer never needs a map.

Option D — Monorepo with packages/ Workspace tooling

npm workspaces enforce layer boundaries at compile time. web cannot import server because workspaces only expose shared-*.
packages/
├── web/              # Next.js frontend, own package.json
├── server/           # GraphQL backend, own package.json
├── shared-types/     # Interfaces, enums, DTOs
└── shared-utils/     # helpers used by both sides

infra/ docs/
package.json    # root workspace manifest
Cost~3–5 days: separate package.json per package, workspaces field, tsconfig paths, CI updates. Sets up for future pnpm / Turborepo.
Trade-offBiggest lift. Overkill unless a third surface (mobile, separate admin SPA) is joining.
Good forHard architectural enforcement — layer violations become compile errors.

Pragmatic path: Option A this week (zero config risk, fastest legibility gain), then Option C later as a single follow-up PR. Option A's tree fits inside Option C cleanly, so no work is wasted. Skip D unless a second app/package is imminent.
