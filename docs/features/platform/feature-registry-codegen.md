# Feature registry — codegen-backed explicit imports

Status: Planned
Last updated: 2026-04-29
Related: [service-modularity.md](service-modularity.md), [plug-and-play-features.md](plug-and-play-features.md)

## What it is

Pick the discovery mechanism for the `feature.manifest.ts` files defined in [service-modularity.md](service-modularity.md). The constraint that drives the design: **every class and service inside a feature folder must remain navigable via WebStorm / VS Code "Find Usages."** That rules out runtime registration patterns and forces a build-time codegen step that emits explicit static imports.

## The problem

JetBrains IDEs (and the TypeScript language service that VS Code uses) index *static* `import` references. "Find Usages" walks that index. Three classes of registry break the index in different ways:

- **`import.meta.glob` / dynamic-import-vars** — The bundler resolves the glob at build time, but the IDE's symbol index never sees a reference to the imported file. Result: classes inside a globbed folder appear unused. Vite-only and non-portable to plain `tsc` / Node ESM. ([JetBrains: Search for usages](https://www.jetbrains.com/help/idea/find-highlight-usages.html), [Qwik: import.meta.glob caveats](https://qwik.dev/docs/cookbook/glob-import/))
- **Runtime filesystem scan + dynamic `import()`** — Same problem at IDE level *and* breaks bundlers. The classes look unused everywhere.
- **Barrel re-exports** (`features/index.ts` that re-exports every feature) — Even with static imports, an over-aggressive barrel layer can hide real consumers from "Find Usages" in JetBrains IDEs. ([JetBrains forum: Find Usages and barrel re-exports](https://intellij-support.jetbrains.com/hc/en-us/community/posts/4406269519506-Getting-better-at-finding-usages))
- **Decorator-driven DI** (`tsyringe`, NestJS-style `@Module({providers: [...]})`) — Works for "Find Usages" because the class still appears in a decorator argument list, but introduces a runtime DI container that's overkill for our footprint. ([NestJS modules](https://docs.nestjs.com/modules))

The pattern that wins on all three axes (IDE traceability, runtime simplicity, ergonomics for adding a feature) is **codegen of explicit static imports**.

## The design

### One generated file

`services/infra/featureRegistry.generated.ts` — committed to git, regenerated by a script. Looks exactly like a hand-written explicit list:

```ts
// AUTO-GENERATED by `npm run features:codegen` — do not edit by hand.
// Edit `services/features/<X>/feature.manifest.ts` and re-run codegen.
import {auditFeature} from '../features/Audit/feature.manifest';
import {cartFeature} from '../features/Cart/feature.manifest';
import {checkoutFeature} from '../features/Orders/feature.manifest';
import {customerAuthFeature} from '../features/CustomerAuth/feature.manifest';
import {inventoryFeature} from '../features/Inventory/feature.manifest';
import {mcpAdminFeature} from '../features/McpAdmin/feature.manifest';
import {productsFeature} from '../features/Products/feature.manifest';
// … one line per feature

import type {FeatureManifest} from './featureManifest';

export const featureRegistry: readonly FeatureManifest[] = [
    auditFeature,
    cartFeature,
    checkoutFeature,
    customerAuthFeature,
    inventoryFeature,
    mcpAdminFeature,
    productsFeature,
];
```

This is what the rest of the system consumes. Because every entry is a real static `import`, WebStorm's index sees the reference; "Find Usages" on `CartService` walks all the way back through `cartFeature.services` → `featureRegistry` → `MongoDBConnection.setupClient` → every resolver that calls it.

### The codegen script

`tools/codegen-feature-registry.ts`. Single-file `tsx` script. Logic:

1. Walk `services/features/*/feature.manifest.ts` in sorted order.
2. Parse each file just enough to read the named export (TypeScript compiler API or simple regex on `export const \w+Feature`).
3. Validate: every manifest exports exactly one `*Feature` const, and the const name matches the folder convention (`<lowerFirstFolderName>Feature`).
4. Emit `featureRegistry.generated.ts` with the imports + the array, sorted alphabetically.
5. Run prettier on the output so diffs are stable.

Wired in `package.json`:

```jsonc
"scripts": {
    "features:codegen": "tsx tools/codegen-feature-registry.ts",
    "features:codegen:check": "tsx tools/codegen-feature-registry.ts --check",   // CI: fail if file is stale
    "predev": "npm run features:codegen && npm run dev:db",
    "prebuild": "npm run features:codegen"
}
```

Pre-dev and pre-build hooks keep the file in sync; CI runs `:check` mode (regenerate to a temp file, diff against the committed file, fail if different) so a missed regen surfaces in PR review, not at runtime.

### Why generate vs hand-maintain

A hand-maintained list works too — it's still explicit static imports, and IDE navigation works. The difference is the cost of adding a feature:

- **Hand-maintained:** drop a manifest, then remember to add an `import` + array entry to `featureRegistry.ts`. Easy to forget; reviewer has to spot it.
- **Generated:** drop a manifest, run `npm run features:codegen`, commit. Or just `npm run dev` and the predev hook does it.

Both are acceptable; generated is recommended because it scales to dozens of features and removes a class of "I forgot to register it" bugs without sacrificing the IDE properties. ([Apollo: modularizing GraphQL schema](https://www.apollographql.com/blog/backend/schema-design/modularizing-your-graphql-schema-code/) recommends the same explicit-imports approach for resolver merging.)

### What the IDE sees

After codegen, "Find Usages" on any feature class returns the full callgraph:

1. `cartService.addItem` → resolver `Mutation.cartAddItem` (in `Cart/graphql/resolvers.ts`)
2. → `cartFeature.resolvers` (in `Cart/feature.manifest.ts`)
3. → `featureRegistry[1]` (in `featureRegistry.generated.ts`)
4. → `featureRegistry` import in `mongoDBConnection.ts`, `graphqlResolvers.ts`, `authzCompose.ts`
5. → every consumer of those files

WebStorm's *Module Dependency Diagram* (`Diagrams → Show Diagram` on a feature folder) gives a visual graph of who imports the feature, including the registry edge. ([WebStorm: Module Dependency Diagrams](https://www.jetbrains.com/help/webstorm/module-dependency-diagram.html))

### Schema and resolver merging stays explicit too

Inside each feature's manifest, the SDL string and resolver tree are also imported statically — no glob there either:

```ts
// services/features/Cart/feature.manifest.ts
import {CartService} from './CartService';
import {cartSdl} from './graphql/cart.sdl';                 // export const cartSdl = `type Cart {...}`
import {cartResolvers} from './graphql/resolvers';
import {cartAuthz} from './authz';
```

Resolver maps are merged with `mergeResolvers` from `@graphql-tools/merge` (already a transitively-available dep). SDL strings are concatenated and parsed by `makeExecutableSchema`. ([GraphQL Tools: Schema Merging](https://the-guild.dev/graphql/tools/docs/schema-merging))

## Comparison

| Approach | "Find Usages" works | Runtime cost | Add-a-feature ergonomics | Bundler portability |
|---|---|---|---|---|
| **Codegen explicit imports (this spec)** | ✅ full | none | one command | any |
| Hand-maintained explicit list | ✅ full | none | one manual edit | any |
| `import.meta.glob` eager | ❌ IDE blind | none | zero | Vite-only |
| Runtime fs scan + `import()` | ❌ IDE blind | per-boot scan | zero | Node-only |
| Decorator DI (NestJS / tsyringe) | ✅ via decorator | DI container | one decorator | any |
| Barrel re-exports | ⚠️ broken in JetBrains | none | one re-export | any |

The codegen path is the only option that's both **IDE-traceable** and **add-a-feature-cheap**. ([Moderne: how Find Usages indexing actually works](https://www.moderne.ai/blog/introducing-the-moderne-ide-plugin-for-jetbrains-intellij-idea))

## Implementation order

This spec is the *mechanism* for [service-modularity.md](service-modularity.md)'s manifest design. They land together:

1. Define `FeatureManifest` interface in `services/infra/featureManifest.ts`.
2. Write `tools/codegen-feature-registry.ts` and the npm scripts.
3. Add the `:check` mode to CI (`.github/workflows/ci.yml`).
4. Migrate the first feature (Audit, smallest) to the manifest pattern. Codegen runs. `featureRegistry.generated.ts` lands with one entry. Build green.
5. Migrate the rest one at a time per the order in `service-modularity.md`.

At every step, "Find Usages" on any service class returns the same set of references it would for a hand-imported class — that's the success criterion.

## Open questions

1. **Commit the generated file or gitignore it?** Recommendation: **commit**. Diff is small, fresh checkouts work without a build step, and the file appearing in PRs makes "I added a feature" reviewable. The CI `:check` keeps it honest.
2. **Codegen with TypeScript compiler API or regex?** Regex is enough for parsing a single `export const xFeature = ...` line and is dependency-free. Switch to the compiler API only if the manifests grow (e.g. multi-export). Recommendation: regex for v1.
3. **Conditional features** — should the registry have all features or only enabled ones (per [plug-and-play-features.md](plug-and-play-features.md))? Recommendation: **all features in the generated registry; the toggle layer reads `featureFlags` and disables them at runtime.** This keeps the static graph complete (so disabled-but-installed features still pop up in "Find Usages") and matches the principle that disable ≠ delete.
4. **Per-feature glob inside a manifest** — e.g. registering all sample-content variants under `services/features/CmsAi/handlers/scaffold/samples/*.ts`. The same rule applies: the manifest should statically import each sample file rather than glob the folder. If the sample list grows large, the same codegen pattern can produce a per-feature `samples.generated.ts`. Worry about it when there are >20 samples.
5. **Migration support for existing static imports** — `mongoDBConnection.ts` and `graphqlResolvers.ts` will lose their direct imports of feature services as features migrate. The codegen also needs to leave a deprecation comment on the legacy imports until the last feature is migrated, so half-migrated states are obvious.
