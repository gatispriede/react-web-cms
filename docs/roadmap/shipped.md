## 2026-05-16 — admin-content-releases (Release entity + atomic publish)

| # | Notes |

| **admin-content-releases — first-class `Release` + atomic publish + rollback + scheduled publish** | Closes the XL [admin/admin-content-releases.md](admin/admin-content-releases.md) roadmap row — the 2025 Sanity differentiator vs Strapi / Payload / Contentful. **Schema:** `shared/types/IRelease.ts` defines `IRelease` + `ReleaseMember` + `ReleaseStatus` (`draft` → `publishing` → `published` / `failed` / `rolled-back`) + `IReleaseSummary` projection. Each member freezes BOTH the draft snapshot (what publishes) AND the pre-release live snapshot (what rollback restores) inline on the release document — no separate `ReleaseSnapshots` collection in v1 to keep the happy-path query shape simple. **Service** (`services/features/Releases/ReleaseService.ts`): CRUD + `attach` / `detach` + `previewAt` (returns members + overlay map `${entity}:${id}` → snapshot) + `resolvePerspective` (single-entity overlay read) + `publish` (atomic) + `rollback` (creates "Rollback of <title>" release that re-attaches reverse snapshots, auto-publishes, marks the original `rolled-back`) + `listDue` (scheduler query). Concurrency: `version` OCC counter + status guard; the `publish` path atomically flips `draft|failed → publishing` under an OCC update so a concurrent publish call short-circuits with "release is busy". **Atomicity:** `writeMembers` runs inside `client.startSession().withTransaction(...)` when the topology supports it (`ReplicaSetWithPrimary` / `Sharded` / `LoadBalanced`); falls back to a sequential write with a compensating-saga revert on standalone Mongo (dev / `mongodb-memory-server`) — if any member's write throws, every already-written member is reverted to its pre-release snapshot in reverse order. **Publishers** (`publishers.ts`): one tiny `EntityPublisher` interface per `ReleaseEntityKind` (`page` / `post` / `product` / `theme` / `navigation` / `footer` / `seo`) with `snapshot(id)` / `publish(id, snapshot, session?)` / `label(snapshot)`. `makeSimplePublisher` covers collection-backed entities (`Posts` / `Products` / `Themes` / `Navigation`); `makeSettingPublisher` covers the `SiteSettings` key/value rows (`footer` / `seo`). Adding a new entity kind = one entry in `buildPublishers` + one value in the `ReleaseEntityKind` enum. **Scheduler** (`ReleaseScheduler.ts`): in-process `setInterval` (same pattern as `BackupScheduler`), checks every 60 s for draft releases with `scheduledFor <= now`, publishes via the same service path so transactional / saga semantics apply identically to manual + scheduled publishes. Publish + rollback wrap their promises in `notifyPromise` for loading / success / failure toasts. Every interactive surface carries a `data-testid` (`releases-pane`, `release-create-*`, `release-list-item-<id>`, `release-attach-*`, `release-publish-*`, `release-rollback-*`, `release-delete-*`, `release-member-detach-<entity>-<id>`, `release-last-error`, `release-detail-status`).
