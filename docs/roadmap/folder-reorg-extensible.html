V4 expanded — product-oriented, ui/client/-owns-content

Structured from a developer's perspective — clear separation of concerns. Each top-level folder answers a single "what am I working on?": ui/client/ = public-site render, ui/admin/ = CMS editor, services/ = API + persistence, shared/ = cross-cutting contracts. A content block is split by concern too: display lives in ui/client/modules/, editor in ui/admin/modules/, the schema both sides agree on in shared/types/. Casing convention: lowercase for layer/concern folders (ui/, modules/, features/, shell/); PascalCase for module/feature folders that map 1:1 to a React component (Hero/, Gallery/, TopBar/, Auth/).

ui/client/ (public + mobile + modules)ui/admin/ (CMS)services/shared/
Design aims — developer's perspective
  • One concern per folder — "Where do I go to render a module?" → ui/client/modules/. "Edit it?" → ui/admin/modules/. "Change the section type?" → shared/types/ISection.ts. "Add a resolver?" → services/features/<name>/. Every question has exactly one answer.
  • Self-explanatory names — ui/client/modules/hero/, not components/A123/.
  • File budget ≤ 400 code lines (imports excluded). Split by responsibility.
  • Delete-by-folder — removing a module = drop 2 folders (ui/client/modules/X, ui/admin/modules/X) + unregister it from the registry.
  • Client vs Admin separation — the two sides share the generic ISection / IItem types from shared/types/ but never import each other's code.

client/ — render concern

Public site + mobile. Contains pages/, features/ (read-side, including top-bar / mobile-nav / logo / footer), modules/ (display only), mobile/.

admin/ — edit concern

CMS editor. Contains shell/, features/ (write-side: auth, users, themes, bundle, dialogs…), modules/ (editors only), styles/.

services/ — data concern

GraphQL server. features/ resolvers mirror admin features (incl. blog, scene-3d when they land); infra/ adapters.
ui/client → shared/types ← ui/admin  ·  shared/types → services  ·  ui/client and ui/admin never import each other

Anatomy of a module — by concern

A content block lives in three places, one per concern. Each folder is self-sufficient within its concern.
client/modules/Hero/             # RENDER concern — display only
├── Hero.tsx                     # public-site render
├── Hero.scss
├── Hero.types.ts                # IHero + EHeroStyle, re-exported from shared/types/
├── i18n/en.json                 # strings used by display
└── index.ts                     # registers the display component

admin/modules/Hero/              # EDIT concern — editor only
├── HeroEditor.tsx               # content + style form
├── HeroEditor.scss
├── i18n/en.json                 # strings used by the editor
└── index.ts                     # registers the editor component

# CONTRACT — the existing ISection / IItem in shared/types/ is already the schema.
# Hero-specific fields live inside IItem.content (JSON string) exactly as today.
# Section modules persist via the generic section resolver in services/modules/ — no per-module services folder.
# Features that DO need their own data layer (blog, scene-3d…) live in services/features/ instead.

Import rules — one dependency direction

Everyone reads from shared/. ui/client/ and ui/admin/ never import each other.
FromMay importMust NOT import
ui/client/modules/* shared/types, shared/enums, ui/client/features, ui/client/styles ui/admin/*, other modules' code
ui/client/pages, ui/client/features, ui/client/mobile ui/client/modules/*, shared/* ui/admin/*, services/*
ui/admin/modules/* shared/types, shared/enums, ui/admin/features/dialogs, ui/admin/styles ui/client/*, other modules' code
ui/admin/shell, ui/admin/features ui/admin/modules/*, shared/* ui/client/*, services/*
services/features services/infra, shared/* ui/*
shared/ (all subfolders) nothing (leaf — types + pure utils only) everything else
Enforce with eslint-plugin-import + no-restricted-imports; CI fails the violation, not review.

Extension recipes

Want to add…Touch
A new section module (e.g. "FAQ")ui/client/modules/Faq/ + ui/admin/modules/Faq/ + one registry entry. No services change — section persistence is generic via ISection/IItem.
Blog featureFeature triple: ui/client/features/Blog/ + ui/admin/features/Blog/ + services/features/Blog/. Has its own routes, data model, resolvers.
3D scenes featureui/client/features/Scene3d/ + ui/admin/features/Scene3d/ + services/features/Scene3d/ (for asset metadata).
2D animation featureui/client/features/Animation2d/ + ui/admin/features/Animation2d/. Backend optional.
A new admin capability (e.g. "Webhooks")ui/admin/features/Webhooks/ + services/features/Webhooks/. Client untouched.
Android / mobile appCreate ui/client/mobile/. Imports ui/client/modules/* + ui/client/features/* + shared/*. Admin + services untouched.
An experimentFeature triple under */features/experiments/<name>/, flag-gated. Delete the folder when done.
Ops dashboardCreate ops/. Talks to services/ via GraphQL only — no direct code import.

Migration phases (WebStorm-friendly)

Phase 1 — Pure moves (WebStorm refactor only)
  · Right-click → Refactor → Move. Imports auto-rewrite; git mv preserved.
  · No content edits. Commit per logical group:
      1. services/                    (was src/Server/ + src/Api/)
      2. shared/ types/ enums/ utils/ (was src/Interfaces/ + enums/ + constants/ + helpers/)
      3. ui/client/modules/*         (was src/frontend/components/SectionComponents/* — display side)
      4. ui/admin/modules/*          (was src/frontend/components/Admin/ConfigComponents/* — editor side)
      5. ui/admin/features/*         (was src/frontend/components/Admin/AdminSettings/*)
      6. ui/client/features/         (top-bar, mobile-nav, logo, footer, navigation,
                                       themes runtime, languages, assets)

Phase 2 — Wiring only (minimal edits)
  · Barrel index.ts per module + feature.
  · Registry auto-discovery reads client/modules/*/index.ts.
  · Config path updates: gqty, i18next, Dockerfiles, compose.yaml, sslServer certs.
  · ESLint import-rule enforcement.

Phase 3 — File edits (separate roadmap item, later)
  · Split files > 400 lines by responsibility.
  · Rename files that don't match their export.
  · Extract feature-local helpers out of root shared/.

Key win: ui/client/ is "everything a user (browser or mobile) ever sees"; ui/admin/ is "everything an author uses to write it"; services/ is "everything serving data". Deleting or extracting a module touches one folder under ui/client/modules/.
